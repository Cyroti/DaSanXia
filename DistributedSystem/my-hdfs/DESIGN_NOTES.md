# 理论与工程权衡说明

## 1. 不一致性的层级与严重性（理论视角）

在多副本系统中，不一致可以按严重程度分层：

1. 可见性延迟（短暂陈旧读）
- 不同副本在短时间内看到不同版本
- 常见于异步复制或“写后立刻读从库”

2. Read-Your-Write 违背
- 写入者随后读到旧值，观察不到自己刚写入的数据

3. Monotonic-Read 违背
- 客户端先读到新值，之后却又读到旧值

4. 丢更新 / 历史分叉
- 并发写入没有被统一序列化，副本应用顺序不同
- 会直接破坏业务约束，严重性高

5. 脑裂与永久分歧
- 出现多主或恢复流程错误，形成冲突提交历史
- 严重性最高，通常需要共识协议与严格 fencing 解决

### 理论上的解决路径

1. 最强一致：共识协议（Raft/Paxos）+ Quorum 读写 + 任期/epoch fencing
2. 课程可落地方案：单主 + 同步 pipeline 复制 + 有序应用
3. 长期修复手段：校验和、版本向量、反熵同步、后台修复

## 2. 生产环境中的权衡（性能 vs 一致性）

真实系统通常采用分级策略：

1. 关键数据路径
- 使用同步复制，写确认必须满足全链或法定多数
- 代价是写延迟上升、吞吐下降

2. 高吞吐路径
- 使用异步复制，接受有界陈旧
- 通过后台收敛与修复保证最终一致

3. 读策略分层
- 强一致读：主副本或 quorum 读
- 低延迟读：最近副本 + 陈旧度预算

4. 故障处理机制
- 租约/任期防脑裂
- 幂等写入 ID 防重试重复提交
- 周期性比对（如 Merkle Tree）和自动修复

## 3. 本项目为何同时保留两种模式

1. `SYNC`
- 用于展示“如何避免复制延迟导致的不一致”
- 写成功后副本应一致，便于讨论顺序一致性语义

2. `ASYNC_OBSERVE`
- 用于教学实验，主动暴露不一致窗口
- 便于观察“先不一致、后收敛”的复制过程

这种双模式设计同时满足：

- 理论分析：不一致层级与修复思路
- 工程实践：一致性与性能之间的实际权衡
